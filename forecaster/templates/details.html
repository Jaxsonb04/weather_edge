How it was built

A machine-learning approach to predicting tomorrow's daily high at SFO, from a single concrete question.

Question

Apply machine-learning methods to a target that is concrete, measurable, and easy to explain: tomorrow's highest temperature.

Experiment

Cleaned NOAA station history, engineered features, trained and tuned XGBoost and LSTM models, and compared them on held-out dates.

Result

The LSTM performed better for daily highs because it learns the recent weather sequence; the final predictor still incorporates live professional forecasts.

Automation

The project now runs on AWS Lightsail. A small Ubuntu server refreshes Google Weather, NWS, Open-Meteo, and KSFO observations on a schedule, rebuilds this page, and pushes the static site to GitHub Pages.

Key challenge: defining the target correctly. Predicting a random timestamp differs from predicting the next local calendar-day high, and that alignment is what keeps the evaluation valid.

Automation stack: Lightsail systemd timers fetch the paid Google Weather API within a strict event budget, update the local SQLite archive, run the blend/scoring scripts, and publish the latest dashboard through a GitHub deploy key. Forecast updates run unattended on the server.

Future work: collect more scored refreshes, add more Bay Area station history, and tune the blend weights only after the archive shows which sources are consistently closest to the NWS high.

LSTM vs XGBoost

The comparison matters because XGBoost is a strong tabular baseline. The LSTM only earns the headline because it improves tomorrow's daily-high error on held-out days.

Why the LSTM wins hereThe target is a daily high, so the recent shape of the temperature curve matters. The LSTM reads the 48-hour sequence directly instead of depending only on engineered lag columns.
Why XGBoost still mattersXGBoost is fast, interpretable, and competitive. Its feature-importance plot shows recent temperature history is the strongest tabular signal.
