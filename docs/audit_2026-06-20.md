## Executive verdict

The losses are one coherent failure, not variance:

1. **The forecaster systematically under-predicts warm/hot days.** The LSTM is effectively
   pinned ~66-67F regardless of actual warmth: signed bias **-4.34F** when actual >= 70F,
   **-6.6F** at >= 72F, up to -12F on individual days (e.g. 2026-05-18 actual 79 / lstm 66.8).
   The `ResidualCalibrator` (75% empirical histogram + 25% normal, sigma 4.66F) cannot capture
   the left-skewed warm-day residuals, so it over-states NO probabilities by 20-40 points. The
   0.3-0.4 probability bucket claims 35.2% but observes 14.3% (n=14) — that *is* the 0.210 gap.
2. **The trade engine bets NO on warm bins.** Every booked trade is NO on a 70-78F bin (betting
   the high won't reach those). SFO summer highs keep landing there, so prediction error becomes
   trade losses directly. Today's snapshot: model 67.6F vs Kalshi crowd 69.5F (we are 1.9F cooler).
3. **The cohort safety block does not fire on the worst days** because it keys on the *model's*
   forecast high (~67F = "normal"); genuinely-warm days slip through disguised as normal
   (`risk.py` `temperature_cohort(forecast_high_f)`).
4. **`research` traded that cohort with no upside** (`live` blocked it; `research` did not). The
   rolling de-bias it was "collecting data for" learns from scored next-day forecast rows written
   daily regardless of trades, so the warm/hot trading was pure loss.

### Two prior hypotheses overturned by verification

- **The stop-loss veto fix (`6094f0b`) IS deployed.** The live payload carries the consensus
  block (`consensus.py` exists only at `27da3a2`, which descends from `6094f0b` + `0491dc2`),
  and the open NO positions carry fresh `model_side_probability` (0.87/0.94) and correctly HOLD.
  The whipsaw losses (~$23) were pre-fix (06-16 -> 06-19) and are **sunk, not ongoing**.
  `model_side_probability=None` on *closed* rows is a per-position display artifact, not a dead veto.
- **Most early-closed losers genuinely lost** (correct NO cuts); only ~2 were true whipsaw. The
  exit machinery is right; the *entries* (warm-cohort NO bets) are the problem.

Dropped as false positives: "research inherited live's 5x caps via `**` spread" (config shows
explicit small caps were binding); "veto dead on box / all closed rows None".

## Remediation plan

- **Phase 0 — honesty + stop the bleed (DONE 2026-06-20, branch `fix/phase0-honesty-and-research-warmhot-block`):**
  - Relabelled the Strategy Lab backtest panel as a hold-to-settlement *counterfactual* scored
    against the wrong truth, and surfaced the whole-book realized ROI/hit-rate/P&L beside it.
  - Blocked the warm/hot cohort on the `research` profile (matches `live`). NB this only fires on
    model-forecast-warm days; the disguised-warm case needs the Phase 2 detector.
- **Phase 1 — truth consolidation (DONE 2026-06-20, same branch):**
  - `load_ksfo_daily_highs` ([forecast.py](../trading/sfo_kalshi_quant/forecast.py)) now prefers the
    CLISFO integer and floors the NWS fallback, so every backtest/settle reader uses the same truth
    as the live CLISFO settle path. `settle_paper_orders` ([db.py](../trading/sfo_kalshi_quant/db.py))
    floors its input at the single chokepoint (covers manual `--settlement-high` + the WeatherEdge
    fallback). Ported the `TEMPERATURE(F)` section anchor into
    [settlement.py](../trading/sfo_kalshi_quant/settlement.py). Three regression tests added.
  - Config-drift guard ([sync_to_lightsail.sh](../trading/deploy/aws/sync_to_lightsail.sh)): refuses
    to deploy a dirty `trading/`/`forecaster/` tree (override `ALLOW_DIRTY_DEPLOY=1`) and writes
    `DEPLOYED_SHA.txt` to the box. This stops *new* drift. **Do NOT blind-rsync HEAD** to clear the
    *existing* drift — the box's `strategy_research.py` emits dashboard fields
    (`model_side_probability`, `stop_loss_basis`, `take_profit_basis`, `exit_rule_reason`) that exist
    in no commit; overwriting them would regress the live dashboard. Resolve it the other direction —
    pull the box's code into git first (manual, SSH, you run it). Set
    `LIGHTSAIL_IP` / `LIGHTSAIL_KEY` from the gitignored `.local/lightsail.env`
    (same as the deploy scripts), then:
    ```bash
    REMOTE=ubuntu@"$LIGHTSAIL_IP"
    scp -i "$LIGHTSAIL_KEY" "$REMOTE":/opt/weatheredge/trading/sfo_kalshi_quant/strategy_research.py /tmp/box_strategy_research.py
    git --no-pager diff --no-index trading/sfo_kalshi_quant/strategy_research.py /tmp/box_strategy_research.py
    # reconcile the box-only field additions into HEAD, commit, THEN deploy.
    ```
- **Phase 2 — the real accuracy fix (partially DONE 2026-06-20, branch
  `feat/phase2-actual-warm-and-cohort-calibration`):**
  - **Actual-warm detection — DONE.** The regime gate now blocks on the WARMER of the model
    forecast and the **market-implied consensus high** ([risk.py](../trading/sfo_kalshi_quant/risk.py),
    flag `regime_block_uses_market_implied_high`, default on). This catches the disguised-warm days
    (model says ~67F, crowd/actual 75F) the model-only block in Phase 0 misses. Already wired through
    the live scan's `rank(market_consensus=...)`. +3 tests.
  - **Cohort-stratified / split-normal calibration — INVESTIGATED, REJECTED (evidence-based).** Ran
    the walk-forward `backtest-calibration` harness as an A/B. (a) No existing calibrator knob
    (`empirical_weight` 0.5-0.9, kernel bandwidth, shrinkage) moves warm/hot Brier off ~0.90 — the
    miscalibration is structural, not a tuning problem (the model collapses warm days to ~67F
    predictions, so no reweighting of the residual blend recovers the warm signal). (b) A split-normal
    (right-skewed) error model trades a tiny warm gain (Brier 0.904->0.881, still >> 0.25 on a
    now-BLOCKED cohort) for a **normal-cohort regression** (0.526->0.531) — and normal/cold is where
    trades actually happen now. Net overall change is within noise (0.531 vs 0.531). This reproduces
    the forecaster team's documented "tail-weighting fixes hot but wrecks normal" wall. **The warm tail
    is point-accuracy/data-limited, not calibration-form-limited**, so no calibration-form change ships.
  - **Real remaining accuracy levers (data-side, out of trading scope):** more inland history / a 2nd
    inland station (Livermore); AI-NWP sources (GraphCast/AIFS via Open-Meteo); the gated rolling
    de-bias auto-activating at 30 clean CLISFO days. To make a *market-informed* calibration
    validatable later, persist the consensus implied-high per decision snapshot (the ladder is not
    stored per row today, so a market-informed calibrator cannot be backtested).
- **Phase 3 — hardening:** floor the posterior at source in `probability.py`; narrow
  `except Exception` + log tracebacks; suppress snapshots on Kalshi-fallback ticks; supplement the
  calibration gate with weighted ECE + bootstrap CI.
